# OrbitForge — Browser-Based Spacecraft State Estimation Engine

**Type:** Personal Project
**Categories:** Simulation, Aerospace, WebAssembly
**Stack:** C++17, WebAssembly (Emscripten), TypeScript, WebGL2, Eigen, Chart.js, GoogleTest, Playwright
**GitHub:** https://github.com/aneeshg5/OrbitForge
**Live:** https://orbitforge.pages.dev

## Overview

OrbitForge loads a real satellite's orbital elements (a TLE, Two-Line Element set, pulled live from CelesTrak) and runs three state estimation filters against it side by side: a linear Kalman Filter (KF), an Extended Kalman Filter (EKF), and an Unscented Kalman Filter (UKF). All three filters see the same stream of noisy, simulated sensor measurements and try to recover the satellite's true position, velocity, and attitude. The user can inject faults mid-run (a GPS outlier, a sensor dropout, an unmodeled maneuver) and watch each filter respond differently, or launch a full Monte Carlo campaign of hundreds to thousands of independent runs to statistically validate whether each filter's own uncertainty estimate is actually trustworthy. Every piece of this, the orbital dynamics integrator, all three filters, the sensor models, and the Monte Carlo thread pool, is C++17 compiled to WebAssembly and runs entirely client-side. There is no backend server, no API, and no per-user compute cost; the browser only renders.

## Motivation and the Gap It Fills

Existing orbit-visualization tools (Celestrak's own viewers, most browser satellite trackers) only run SGP4 propagation, they show where a satellite is, not how confidently a filter could know that from noisy measurements. Existing state-estimation toolchains (Orekit, GMAT, MATLAB's EKF/UKF toolboxes) are desktop or server software with no live, interactive, in-browser filter comparison. OrbitForge's premise is that the actually interesting quantity for a GNC (Guidance, Navigation, and Control) engineer is the *estimation error*, not just the propagated position, and that this is something a pure SGP4 propagator cannot show at all. OrbitForge treats the TLE only as an initial condition: once a run starts, the "true" trajectory is propagated by the engine's own RK4 integrator with full perturbation models, which is intentionally a higher-fidelity ground truth than SGP4 itself. Noisy sensor measurements are synthesized from that true trajectory, and the filters never see the true state, only the noisy measurements, exactly mirroring how a real spacecraft's onboard navigation filter only ever sees its own sensors.

## State Estimation Filters

Three filters run simultaneously every tick against identical measurements, implemented as C++ template classes over an abstract `FilterBase` with `predict()`/`update()` virtual methods, all using fixed-size, stack-allocated Eigen matrices (no heap allocation in the hot path).

- **Linear Kalman Filter (KF):** Stays a 6-state filter (position and velocity only) by design, linearizing the dynamics around a fixed reference orbit. This is the deliberately naive baseline; it visibly diverges on a curved orbit because a linear filter cannot track a fundamentally nonlinear two-body trajectory well over time, which is the whole pedagogical point of including it.
- **Extended Kalman Filter (EKF):** Propagates the full 12-state nonlinear dynamics via RK4 and uses analytically derived Jacobians (not finite differencing) for both the orbital block (two-body gravity plus the full J2 Jacobian) and the attitude block.
- **Unscented Kalman Filter (UKF):** Implemented in square-root form (SR-UKF) to guarantee a positive-definite covariance at every step, using a rank-1 Cholesky update/downdate via Givens rotations rather than a full re-factorization. Sigma points are propagated through the same RK4 integrator used for the true trajectory, so no Jacobian is computed anywhere in the UKF, which is its core advantage over the EKF for handling the rigid-body gyroscopic term that has no convenient linearization.

## 6DOF Attitude Estimation (Multiplicative EKF)

Both EKF and UKF were extended from a 6-state orbital-only filter to a full 12-state Multiplicative EKF (MEKF) that jointly estimates orbital state and attitude: `[delta_theta(3), omega(3), r(3), v(3)]`, a 3-component multiplicative attitude error, 3-component body-frame angular velocity, and the original 3-position/3-velocity orbital block. The attitude itself is tracked as a separate reference quaternion `q_ref`, updated by composing in the small-angle attitude-error correction after every measurement update and then resetting the error state to zero, the standard MEKF pattern used in real flight software (Markley and Crassidis) specifically to avoid ever linearizing through a 4-parameter unit-norm quaternion constraint inside the covariance matrix. The true attitude trajectory is propagated by Euler's torque-free rigid-body equation (no reaction wheels, magnetorquers, or external torques modeled, intentionally) together with quaternion kinematics. The KF was deliberately left untouched at 6 states rather than given a parallel, weaker attitude linearization, since rigid-body rotation is more nonlinear than orbital motion and giving the naive baseline a second, separately bad linearization would muddy the comparison.

## Sensor Models and Fault Injection

Four sensor models synthesize the noisy measurements every filter actually sees: a position-only GPS sensor (ECI to ECEF rotation plus Gaussian noise, default sigma 10 meters), an accelerometer with a random-walk bias, a gyroscope with its own random-walk bias that directly measures the angular velocity state, and a magnetometer using an IGRF-13 dipole approximation of Earth's magnetic field that is what actually makes the attitude error observable (the gyroscope alone only constrains angular velocity, not orientation). Faults are injected at the sensor layer, not the physics layer: the true trajectory keeps propagating completely untouched, and only what the filter believes changes, which is the cleanest possible illustration of the difference between what is actually happening and what a filter's covariance thinks is happening. Five fault types are exposed in the UI: a single GPS measurement spike, a timed GPS dropout window, an unmodeled instantaneous velocity maneuver, a persistent drag-coefficient model error, and a persistent additive GPS bias.

## Monte Carlo Consistency Analysis

A Monte Carlo campaign runs N independent realizations of the same scenario, each with an independently sampled noise draw, and computes the time series of RMS position/velocity error alongside two formal statistical consistency metrics: NEES (Normalized Estimation Error Squared, checking whether the filter's actual error stays inside its own predicted covariance bounds) and NIS (Normalized Innovation Squared, the same idea applied to the measurement residual). Both are checked against theoretical chi-squared confidence bounds computed for the campaign's actual run count and the filter's actual state dimension (6 for KF, 12 for EKF/UKF), via a Wilson-Hilferty approximation to the chi-squared quantile function implemented from scratch in C++. A campaign is fully configurable from the UI (run count, filter choice, run duration, process noise, and a fixed or random seed) and is distributed across a 4-thread pool of real OS threads inside the WASM module via Emscripten pthreads. The thread pool reports live progress without blocking the UI: a separate atomic counter incremented by the worker threads is polled directly off the shared WASM heap by the main thread, completely independent of the blocking Monte Carlo call itself.

## Architecture: Thread Model and Data Flow

The simulation, all three filters, and the sensor models run inside a dedicated Web Worker at a fixed 100 Hz tick rate, fully decoupled from render rate. Each tick calls a single `Simulation::step(dt)` function that propagates the true trajectory, runs all three filters' predict and update steps, applies any pending fault, and writes one `StateFrame` struct (true state, all three filters' estimates, covariance diagonals, NIS values, active fault) into a lock-free single-producer single-consumer ring buffer backed by a `SharedArrayBuffer`. The main thread never blocks on the worker: it drains whatever frames are available from the ring buffer once per animation frame (typically 60 fps) and renders them with a custom WebGL2 scene (an Earth sphere with day/night terminator shading, the Sun and Moon on stylized orbits, per-filter colored orbit paths, 3-sigma covariance ellipsoids, and a body-axis attitude gizmo) plus four live Chart.js panels, each independently selectable among more than 20 metrics including position/velocity error, covariance trace, NIS, and attitude/angular-velocity error. `step(dt)` takes its time delta as an explicit argument with zero dependency on wall-clock time, which is what makes the entire simulation core directly unit-testable without any timing flakiness; `start()`/`pause()` are a thin wrapper that calls `step()` repeatedly from a background thread at a fixed cadence.

## Memory Systems

The lock-free ring buffer (`SPSCRingBuffer<StateFrame, 512>`) requires its capacity to be a power of two so index wraparound is a bitmask rather than a modulo, and gives its write and read head pointers their own separate 64-byte-aligned cache lines with explicit padding between them, since without that padding the two indices (written by two different threads) would share a cache line and every push from the producer would invalidate the consumer's cached copy of a line it never actually touched, a textbook false-sharing bug. Memory ordering on the hot path is deliberately the minimum needed: a relaxed load on a thread's own index (no cross-thread synchronization required), an acquire load on the other thread's index, and a release store after writing data, with no sequentially-consistent fences anywhere; in WASM, `std::atomic<size_t>` on a `SharedArrayBuffer`-backed object compiles directly to `Atomics.load`/`Atomics.store` with matching ordering semantics, so the lock-free guarantees carry over from native C++ to the browser unchanged. A full buffer silently drops the newest frame rather than blocking the simulation, since the simulation must never stall waiting on the renderer. A separate fixed-size, cache-line-aligned pool allocator means the simulation tick allocates nothing on the heap once a scenario is initialized: all filter state is stack-allocated fixed-size Eigen matrices, and any per-tick working buffer comes from a pool sized once at scenario init, avoiding the unpredictable allocator-lock latency `malloc` would introduce inside a tight 100 Hz loop. The Monte Carlo ensemble additionally uses a Structure-of-Arrays layout (one contiguous array per state field across all N runs, rather than an array of per-run structs) so the batched RK4 inner loop accesses memory cache-sequentially and is SIMD-vectorizable.

## Performance

Measured natively in a Release build on the development machine (not WASM, which carries some additional overhead): a single EKF predict-plus-update step (12-state, with GPS, gyro, and magnetometer measurements) costs about 1.19 microseconds; a single UKF step in square-root form with 25 sigma points costs about 5.53 microseconds; all three filters running simultaneously for one simulation tick cost about 6.93 microseconds combined, comfortably inside the 10-millisecond budget a 100 Hz tick allows by more than three orders of magnitude. The lock-free ring buffer sustains roughly 525 million single-threaded push/pop pairs per second. The Monte Carlo Structure-of-Arrays layout with a batched, vectorization-friendly kernel measured a roughly 495% throughput improvement over a naive array-of-structs baseline on the same batch RK4 workload, and a full Monte Carlo campaign of 1,000 independent runs over 1,000 steps each (one million total filter updates) across the 4-thread pool completes in under 420 milliseconds.

## Testing and CI

The C++ engine has 112 GoogleTest unit and integration tests covering orbital mechanics (Kepler orbit closure, energy conservation), filter behavior (covariance growth under predict, covariance shrink under update, MEKF reset correctness and quaternion-norm preservation), rigid-body dynamics (angular momentum and kinetic-energy conservation for torque-free motion), the lock-free ring buffer under a real multi-threaded producer/consumer stress test, the pool allocator's alignment and reuse guarantees, and statistical filter consistency across 100-run Monte Carlo campaigns checked against chi-squared NEES bounds. All 112 tests pass clean under both AddressSanitizer/UndefinedBehaviorSanitizer and, separately, ThreadSanitizer. The TypeScript frontend is type-checked with `tsc --noEmit` and covered by 8 Playwright end-to-end browser tests exercising the full run/pause/reset lifecycle, sim-speed changes taking effect correctly across a pause/continue cycle, run-duration auto-stop behavior, and a Monte Carlo campaign completing with real, finite numeric results rather than NaN. GitHub Actions CI runs three jobs on every pull request: a native engine build and test under both sanitizer configurations, a frontend typecheck, and a full Emscripten WASM build followed by the Playwright e2e suite against a real dev server with the actual compiled WASM module loaded. Deployment to the live site is a separate GitHub Actions workflow that builds the WASM bundle and the Vite frontend and pushes the result to Cloudflare Pages.
