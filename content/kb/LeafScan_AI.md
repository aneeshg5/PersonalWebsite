# LeafScan AI — Full-Stack Mobile ML Application

**Type:** Personal Project
**Period:** February 2026 – Present
**Categories:** Mobile, ML/AI, Backend
**Stack:** React Native (Expo), FastAPI, PyTorch, Supabase (PostgreSQL + Auth + Storage), Docker, TypeScript, GitHub Actions
**GitHub:** https://github.com/aneeshg5/LeafScan-AI
**Live Demo:** https://appetize.io/app/b_bcuxan5s6shhj6stu2osaj3kj4
**Live API:** https://leafscan-api-1due.onrender.com (interactive docs at `/docs`)

## Overview

LeafScan is a full-stack mobile app that lets a farmer photograph a plant leaf, get a disease diagnosis from a self-trained convolutional neural network in under a second, and then chat with an AI agronomist for treatment advice grounded in live web search. It was built solo, end to end: the PyTorch model and training pipeline, the FastAPI inference and chat backend, the React Native mobile client, the Postgres schema and Row Level Security policies, and the CI/CD and deployment configuration. The project intentionally goes past "train a classifier and call it done" — it tackles the production concerns that usually get skipped in a portfolio project: out-of-distribution detection, rate limiting without extra infrastructure, GPS-based record deduplication, and a real security boundary between the mobile client and the database.

## Machine Learning: Disease Classifier

The core model is an EfficientNet-B0 (5.3M parameters) fine-tuned via transfer learning on the PlantVillage dataset (~54,000 leaf images, 38 classes spanning apple, corn, grape, potato, tomato, and more, healthy and diseased variants for each). Training was done on Kaggle's free GPU tier: ImageNet-1K pretrained weights with all backbone layers frozen, only the replaced 1280-to-38 classifier head trained with Adam at a 1e-4 learning rate, over 10 epochs with random horizontal flip, rotation, and color jitter augmentation. The result is ~97% validation accuracy. At inference time the model runs CPU-only on a Render free-tier instance and still returns a classification in under 200ms, which matters because the project deliberately avoided paying for GPU hosting — proving the model architecture choice (EfficientNet-B0 over a heavier backbone) was the right inference-per-cost tradeoff for a real deployment constraint, not just a notebook benchmark.

## Out-of-Distribution Detection (Mahalanobis Distance)

A plain softmax classifier will confidently misclassify an image of a hand, a wall, or a dog as one of its 38 known leaf diseases, because softmax always produces a normalized probability distribution even on inputs it has never seen anything like. To catch this failure mode before it reaches the user, the backend extracts the 1280-dimensional backbone embedding from EfficientNet's pre-classifier layer for every incoming image and computes the minimum Mahalanobis distance to 38 per-class centroids fit on the full training set, using a tied covariance matrix with a small regularization term for numerical stability. If that distance exceeds a calibrated threshold (tuned to the 99th percentile of training-set scores, configurable via an environment variable), the request is rejected with HTTP 422 before it ever reaches the classifier head and before it counts against the user's daily scan quota. The class means and precision matrix are precomputed offline by a dedicated script run against the training set and shipped as `.npy` artifacts; if those artifacts are ever missing, the check degrades gracefully and the original behavior is preserved rather than crashing the service. This is the single most ML-engineering-flavored piece of the project: it required understanding why softmax probabilities are not confidence, and implementing a distance-based detector in the model's actual feature space rather than relying on a token "confidence threshold" heuristic.

## Backend: FastAPI Inference and Chat Server

The backend is a Python 3.11 / FastAPI service, containerized with Docker and deployed to Render. `POST /predict` accepts a multipart image upload, runs it through the OOD check and then the classifier, uploads the original image to Supabase Storage, writes a structured scan row to Postgres, and returns disease name, confidence, severity, and a treatment list, all behind JWT authentication. Per-user rate limiting (50 scans/day, configurable) is implemented as a rolling 24-hour window: instead of adding Redis or new columns, it counts existing rows in the `scans` table filtered by `user_id` and `created_at`, which resets automatically as old rows age out of the window. A second endpoint, `POST /drone/scan`, accepts the same image plus GPS coordinates authenticated by a per-user API key, letting an external script or a drone submit scans autonomously without a human in the loop — coordinates are matched to the nearest existing plant record via the Haversine formula within a 10-meter radius, auto-creating a new plant record on a miss. A `GET /history` endpoint serves a paginated, soft-deletable scan log per plant. The whole app is tested with pytest in CI using a lazy Supabase client proxy that defers `create_client()` until the first real database call, so the health-check and predict tests run against placeholder environment variables without ever needing a live database connection.

## AI Agronomist: RAG Chat Pipeline

Each plant has its own persistent chat thread backed by GPT-4o-mini. The pipeline is deliberately cost-aware: a frozenset of 60+ agricultural keywords gates whether a Tavily web-search call fires at all, so a purely conversational message like "thanks, got it" skips the search entirely rather than paying 500-1500ms of latency and an API call for a turn that doesn't need grounding, an estimated 70% reduction in search calls. When search does fire, the model's reply is grounded in real Tavily results and the response includes clickable source links. Product names mentioned in the reply are extracted with a regex pass and used to kick off a concurrent Serper Google Shopping search, surfacing real purchasable treatment products with price and store. After each session, a second, capped GPT-4o-mini call (max 200 tokens) extracts one to three durable observations about the plant and writes them to a `plant_memories` table; future sessions prepend those facts to the system prompt, so the agronomist remembers prior context about a specific plant across conversations rather than starting cold every time.

## Mobile App: React Native / Expo

The client is built on Expo SDK 54 with expo-router v6 for file-system based, typed navigation, and nativewind v2 for Tailwind-style styling compiled to native StyleSheet at build time. The scan screen uses the device accelerometer (polled at 80ms intervals, a 0.06g stillness threshold held for 600ms) to detect when the phone is steady enough to capture a sharp leaf photo, plus pinch-to-zoom and a drag slider mapped onto the camera's native zoom range. After capture, the app resolves device GPS and compares it against the user's existing plant records to decide whether to link the scan to an existing plant or prompt to create a new one. A `resultCache` in-memory map hands the freshly classified result straight to the result screen without a second network round-trip, falling back to a database fetch only if the app process was restarted. Auth tokens are persisted through a custom chunked SecureStore adapter, written specifically to work around a roughly 1800-character per-key storage limit on some Android versions that a raw Supabase JWT + refresh token pair can exceed; the adapter splits the value across `${key}_0`, `${key}_1`, etc. transparently to the Supabase JS client. The map tab uses `react-native-maps` in a hybrid Google Maps view with color-coded healthy/diseased pins, and ships a web-platform stub (via a custom Metro `resolveRequest`) since the native maps module otherwise crashes the Expo web bundler.

## Security and Data Architecture

The mobile bundle ships only the Supabase anonymous key, which is safe to expose because every table has Row Level Security enabled and scoped to `auth.uid()` — the client can never read or write data it doesn't own, regardless of what the app code does. The Supabase service-role key, which bypasses RLS entirely, lives only in backend environment variables and is never sent to a client. For the drone integration, API keys are generated as `"lscan_" + secrets.token_hex(32)` (256 bits of entropy), and only a SHA-256 hash of the key is ever stored in the database — the raw key is shown exactly once at creation time, so a database compromise yields unusable hashes rather than working credentials. Keys are soft-revoked rather than deleted, preserving an audit trail of `last_used_at` for each integration.

## DevOps: CI/CD and Deployment

GitHub Actions runs two parallel jobs on every push: a backend job that lints with Ruff and runs the pytest suite against placeholder Supabase credentials, and a mobile job that runs `tsc --noEmit` for full TypeScript type checking, both on Node 20 (required by React Native 0.81's minimum version). The backend deploys as a Docker container to Render's free tier. Android builds run through EAS (Expo Application Services) cloud builds, deliberately restricted to the `arm64-v8a` and `x86_64` ABIs with an increased JVM heap, a fix discovered after the default four-ABI build configuration exhausted the ~6GB RAM ceiling on Render's free Gradle worker; `arm64-v8a` covers real Android phones and `x86_64` covers the Appetize.io browser-based emulator used for the live demo link. A web build target (via `expo export --platform web`) is also hosted on Vercel for browsers that can't run an APK directly.
